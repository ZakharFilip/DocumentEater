# Анализ парсеров и соответствия колонок

## 1. Соответствие столбцов таблицы и массива второго модуля

Цепочка данных проверена:

- **GeneralParser.MakeRow** формирует список из 11 элементов **строго в порядке** `TableColumns.Headers`: График, Тип заявки, Приоритет, Кол-во, Условие, SLA, SLA повышенная нагрузка, ЗИП-И, ЗИП-П, ЗИП-З, Услуги.
- **SemanticMarkupService** при создании записи делает `fields[headers[i]] = row[i]` (и при нехватке элементов в `row` подставляет "—"). Порядок полей в записи совпадает с порядком колонок.
- **ResultTablePageViewModel.BuildColumnNames** строит список колонок: сначала все ключи из `TableColumns.Headers` (в том же порядке), затем остальные ключи. **BuildRows** заполняет `Values[i]` как `record.Fields[columnNames[i]]`, поэтому порядок значений в строке таблицы совпадает с порядком заголовков.

Итог: при корректной работе парсеров данные в ячейках должны соответствовать заголовкам. Если столбцы «перепутаны», причина в том, что парсеры записывают данные не в те поля контекста (см. раздел 2).

## 2. Логический анализ парсеров

### Использование AllTables

| Парсер | allText | AllTables |
|--------|---------|-----------|
| FindAppendixTablesParser | да (блоки приложений) | да (регистрация таблиц) |
| WorkSheduleParser | да, затем fallback по всему тексту | да (поиск графика в ячейках) |
| BidParser | да | да (типы заявок в ячейках) |
| PrioritetParser | да | да |
| QuantityParser | да | да (сроки + приоритет/тип в строке) |
| ConditionParser | да | да (фразы условий в строках таблиц) |
| SlaParser | да | да (таблицы ЗИП, маркеры повышенной нагрузки) |
| ZipParser | да (fallback «см. текст») | да (основной источник колонок ЗИП) |
| ServicesParser | да (Приложение2/3, затем весь текст) | да (нумерованные пункты в ячейках) |

### FindAppendixTablesParser
- Находит блоки «Приложение № N» в `allText`, сохраняет диапазоны **по номерам строк** (Start, End).
- Для таблиц из `AllTables` сохраняет «ТаблицаK» → (индекс таблицы, число строк). Табличные ключи не используются как диапазоны строк.

### WorkSheduleParser
- Ищет график: 1) в блоках Приложение2/4/6 по ключевым фразам; 2) по **всему allText**; 3) во **всех ячейках AllTables**. Расширены шаблоны часов (рабочие дни, пн–пт, 9–18 и т.д.).

### BidParser
- Ищет типы заявок в **allText** и во **всех ячейках AllTables**. Добавлен гибкий шаблон «Перечень … работ», контекст для «Обслуживание» (заявки/категории/работы), чтобы не ловить «Календарь обслуживания».

### PrioritetParser
- Собирает приоритеты по тексту и по **AllTables** (каждая ячейка).

### QuantityParser
- Текст: маркеры разделов + привязка по строке. **AllTables**: в каждой строке таблицы ищется число+единицы и приоритет/тип заявки в той же строке; при отсутствии — ключ «—». Добавлены маркеры («срок выполнения», «реакция», «норматив»).

### ConditionParser
- Ищет фразы условий в **allText** и в **строках AllTables**. Расширен список фраз («по запросу», «по согласованию сторон» и т.д.).

### SlaParser
- SLA из таблиц с ЗИП/ККТ. **SLA повышенная нагрузка**: поиск маркеров в тексте и в **тексте всех таблиц AllTables**.

### ZipParser
- Основной источник — **AllTables** (заголовок с ЗИП/Правила установки/комплект). Поиск колонок по вариантам «ЗИП-И», «ЗИП И» и т.д. Fallback по allText — «см. текст».

### ServicesParser
- Приложение2/3 в allText, при отсутствии — поиск нумерованных пунктов по **всему тексту** и в **ячейках AllTables** (регулярка для 1.1, 2.1 и т.д.).

## 3. Может ли код парсеров привести к нужному результату

- **Да**, при условии что:
  1. Документ содержит ожидаемые формулировки (ключевые фразы, заголовки, структура приложений и таблиц).
  2. Порядок полей в контексте и в итоговой строке не меняется (это зафиксировано в `GeneralParser.MakeRow` и `SemanticMarkupService`).

- **Ограничения:**
  - BidParser может добавлять лишние типы заявок из случайных вхождений слов.
  - FindAppendixTablesParser для таблиц хранит (индекс, число строк), а не диапазон строк; парсеры, которые используют только имена приложений, от этого не страдают.
  - Если в документе нет ожидаемых маркеров или структура другая, соответствующие поля останутся "—".

Рекомендация: при «перепутанных» данных проверить, какие именно поля контекста заполняют парсеры (WorkSchedule, BidTypes, Priorities, QuantitiesByKey, Condition, Sla, SlaHighLoad, ZipI, ZipP, ZipZ, Services) и сравнить с порядком в `TableColumns.Headers` и в `MakeRow`.
